# SPEC-006 — Async Bridging Strategy

**Status:** Accepted  
**Version:** 1.0.0  
**Date:** 2026-04-02  
**Authors:** Sarah Project  
**Supersedes:** —  
**Related:** SPEC-001, SPEC-002, SPEC-004, ADR-006

---

## 1. Problem Statement

Swift `async func` declarations must be representable in the transpiled Rust Core and re-exported through the UniFFI boundary as a native Swift `async func` in the generated Shell. Three competing strategies exist:

1. **Native async** — Rust `async fn` exported directly via `#[uniffi::export]`, generating a native Swift `async func`.
2. **Callback bridge** — Rust spawns a Tokio task internally and delivers the result via a UniFFI callback interface; the Swift Shell wraps the call in `withCheckedThrowingContinuation`.
3. **Hybrid (three-zone model)** — Rust Core owns the executor entirely; the UniFFI boundary uses the stable callback pattern; the Swift Shell presents a pure `async func` surface generated by the transpiler.

This specification adopts **Strategy 3** and defines the rules for its implementation.

---

## 2. Motivation

Strategy 1 (native async) offers the best surface ergonomics but currently carries a known `Sendable` conformance gap in UniFFI's generated Swift bindings (upstream issue open as of UniFFI 0.28). This gap causes spurious warnings under `-strict-concurrency=complete` and is not yet resolved.

Strategy 2 (callback-only) solves the `Sendable` gap but breaks Swift structured concurrency, makes cancellation manual, and produces a Swift API that feels like pre-`async`/`await` era code.

Strategy 3 resolves both problems: the `Sendable` gap never surfaces because the FFI boundary is synchronous; Swift structured concurrency is fully preserved because the Shell presents a native `async func` surface; cancellation is automatable by the transpiler.

## 3. The Three-Zone Model

```
┌─────────────────────────────────────────────────────┐
│  SwiftUI Layer  (hand-written)                      │
│                                                     │
│  await viewModel.loadUser(id: id)                   │
│  try await viewModel.saveDocument(doc)              │
└───────────────────────┬─────────────────────────────┘
                        │  native Swift async func
                        │  (generated by Sarah)
┌───────────────────────▼─────────────────────────────┐
│  Shell ViewModel  (Sarah-generated)                 │
│                                                     │
│  func loadUser(id: String) async throws -> User {   │
│    try await withCheckedThrowingContinuation { k in  │
│      CoreFFI.loadUser(id: id, callback:             │
│        ResultCallback { result in k.resume(with: result) })
│    }                                                │
│  }                                                  │
└───────────────────────┬─────────────────────────────┘
                        │  UniFFI callback interface
                        │  (stable, Sendable-gap-free)
┌───────────────────────▼─────────────────────────────┐
│  Rust Core  (Sarah-transpiled + hand-written)       │
│                                                     │
│  fn load_user(id: String,                          │
│               callback: Box<dyn FetchCallback>) {  │
│    tokio::spawn(async move {                        │
│      let result = do_load_user(id).await;           │
│      callback.on_result(result);                    │
│    });                                              │
│  }                                                  │
└─────────────────────────────────────────────────────┘
```

### Zone Responsibilities

| Zone | Owner | Async model | Generated? |
|------|-------|------------|------------|
| SwiftUI Layer | App developer (hand-written) | Swift `async`/`await` | No |
| Shell ViewModel | Sarah transpiler | `withCheckedThrowingContinuation` wrapping | **Yes** |
| Rust Core | Sarah transpiler + developer | Tokio `async fn`, `spawn_blocking` | Partially |

## 4. Async Tier Classification

SPEC-001's classifier is extended with two async tiers:

| Tier | Swift source pattern | Rust Core output | Shell output | Diagnostic |
|------|---------------------|-----------------|--------------|------------|
| **A1** | `async func f() throws -> T` | `fn f(callback: Box<dyn Callback<T>>)` + Tokio spawn | `withCheckedThrowingContinuation` wrapper | — |
| **A1-sync** | `async func f() -> T` (no throws) | `fn f(callback: Box<dyn Callback<T>>)` + Tokio spawn | `withCheckedContinuation` wrapper | — |
| **A2** | `async func` on a `class` receiver | `async fn` on `Arc<Mutex<T>>` | `withCheckedThrowingContinuation` + lock | `ASYNC-LOCK-RISK` |

### Tier A2 — `ASYNC-LOCK-RISK` Diagnostic

When the classifier detects an `async func` whose receiver is a `class` (Tier 2 `Arc<Mutex<T>>`), it emits:

```
ASYNC-LOCK-RISK  [A2]
  Async function on a class receiver detected.
  Holding a Mutex across an await point causes deadlock.
  → Extract async logic into a free function that accepts
    value-type arguments cloned from the locked state.
  → See SPEC-003 §5 for the recommended refactor pattern.
```

Generation proceeds but the emitted Rust code wraps the lock acquisition and release explicitly around non-async work, releasing the lock before the first `.await` point:

```rust
// TIER-A2: lock released before await — ASYNC-LOCK-RISK mitigated by codegen
pub async fn refresh(self: Arc<Self>) {
    let snapshot = {
        let guard = self.inner.lock().unwrap();
        guard.clone() // clone out what you need
    }; // lock released here
    let result = do_network_call(snapshot).await;
    let mut guard = self.inner.lock().unwrap();
    guard.apply(result);
}
```

## 5. `spawn_blocking` Preference for Tier 1 Logic

For Swift `async func` calls whose Rust implementation is purely synchronous (file I/O, parsing, computation), the generated Rust output uses `tokio::task::spawn_blocking` rather than `async fn`:

```rust
// Input: Swift async func parseDocument(_ data: Data) -> Document
// Output: Tier A1 with spawn_blocking
pub fn parse_document(data: Vec<u8>, callback: Box<dyn ParseCallback>) {
    tokio::spawn(async move {
        let result = tokio::task::spawn_blocking(move || {
            // Pure sync Rust — no async contamination
            // Tier 1 golden-core code here
            parse_document_sync(&data)
        }).await;
        callback.on_result(result.unwrap());
    });
}
```

This keeps the transpiled Core logic in golden-core safe Rust with no `async` keyword, preserving compatibility with formal verification tools and making the code easier to audit.

**Rule:** If the Swift source function's body contains no `await` expressions (i.e. it is only declared `async` for caller convenience), the transpiler MUST emit `spawn_blocking` rather than an `async fn`.

## 6. Cancellation

The transpiler generates cancellation support for Tier A1 functions annotated with `@cancellable` (a Sarah-specific Swift annotation):

```swift
// Swift source
@cancellable
async func fetchFeed() throws -> [FeedItem]
```

Generated Swift Shell:

```swift
func fetchFeed() async throws -> [FeedItem] {
    let token = CancellationToken()
    return try await withTaskCancellationHandler {
        try await withCheckedThrowingContinuation { k in
            CoreFFI.fetchFeed(token: token, callback:
                ResultCallback { result in k.resume(with: result) })
        }
    } onCancel: {
        token.cancel()
    }
}
```

Generated Rust Core:

```rust
pub fn fetch_feed(
    token: Arc<CancellationToken>,
    callback: Box<dyn FeedCallback>,
) {
    tokio::spawn(async move {
        tokio::select! {
            result = do_fetch_feed() => callback.on_result(result),
            _ = token.cancelled() => callback.on_cancelled(),
        }
    });
}
```

## 7. Migration Path to Native Async

The three-zone model is explicitly designed as a staged approach. The migration requires changes to **one layer only** — the generated Shell bridge — and leaves Rust Core and SwiftUI layer untouched:

| Stage | When | Shell bridge | Rust Core |
|-------|------|-------------|----------|
| **Stage 1 (current)** | Now | `withCheckedThrowingContinuation` | Callback + Tokio spawn |
| **Stage 2** | When UniFFI Sendable gap resolves | Native `async func` (generated) | `async fn` + `#[uniffi::export]` |

The transition is gated on the UniFFI upstream Sendable issue closing. Sarah's transpiler will expose a `--async-mode=[bridge|native]` flag; `bridge` is the default until Stage 2 is declared stable.

---

## 8. Invariants

1. **No Tokio runtime code in the Shell.** The Shell must never import or reference `tokio` directly.
2. **No raw `Future` types across the UniFFI boundary.** All async results cross the boundary as concrete values delivered to a callback.
3. **Locks are never held across an `await` point.** The `ASYNC-LOCK-RISK` diagnostic enforces this.
4. **`spawn_blocking` for all sync-body async functions.** No async keyword in generated Tier 1 Core code.
5. **The Swift surface is always native `async func`.** Regardless of the internal bridge strategy, the SwiftUI layer sees only idiomatic Swift async APIs.
