# SPEC-006 — Async Strategy Evaluation & Recommendation

**Status:** Accepted  
**Phase:** 2b (async scaffold generation)  
**Authors:** Sarah Architecture Working Group  
**Date:** 2026-04-02  
**Supersedes:** —  
**Related:** SPEC-004 (UniFFI boundary generation), ADR-005 (tiered lowering)

---

## 1. Context

Swift and Rust each have a mature, idiomatic async model:

| Side | Model | Executor | Cancellation |
|------|-------|----------|--------------|
| Rust Core | `async`/`await` + `Future` | **Tokio** (owned entirely by Core) | `tokio::CancellationToken` |
| Swift Shell | `async`/`await` + `Task` | **Swift Concurrency runtime** | `Task.cancel()` / `withTaskCancellationHandler` |

The FFI boundary between them is managed by **UniFFI**. The central architectural question is: *how should Sarah generate the seam between these two worlds?*

This document evaluates all viable strategies, ranks them for suitability in a **transpiler context**, identifies their downsides, and recommends the canonical approach.

---

## 2. Strategy Catalogue

### Strategy A — Three-Zone Hybrid (Tokio | Callback Bridge | Swift async shell)

```
┌───────────────────────┐
│  Swift Shell          │  async func loadData() async throws → T
│  (SwiftUI / ViewModel)│  uses withCheckedThrowingContinuation
└──────────┬────────────┘
           │  UniFFI-generated call — synchronous from UniFFI's perspective
┌──────────▼────────────┐
│  Callback Bridge      │  uniffi::export(callback_interface)
│  (generated by Sarah) │  trait LoadDataCallback: Send + Sync {
│                       │      fn on_result(&self, result: T);
│                       │      fn on_error (&self, error: String);
│                       │  }
└──────────┬────────────┘
           │  Arc<dyn Trait> crossing FFI
┌──────────▼────────────┐
│  Rust Core            │  tokio::spawn(async move { ... })
│  (Tokio executor)     │  Full Tokio ecosystem available
└───────────────────────┘
```

**How it works:**  
The Rust function accepts an `Arc<dyn FooCallback>` and immediately spawns a Tokio future. When the future resolves it calls `callback.on_result(value)` or `callback.on_error(msg)`. The Swift Shell wraps each generated function with `withCheckedThrowingContinuation`, bridging the callback back into a native Swift `async func`.

**Mechanism:** UniFFI `callback_interface` — stable, well-documented, Sendable-gap-free in current releases.

---

### Strategy B — Native UniFFI Async End-to-End

```
┌───────────────────────┐
│  Swift Shell          │  let result = try await coreLoad()  ← generated binding
└──────────┬────────────┘
           │  UniFFI async scaffolding (RustFuture lifecycle)
┌──────────▼────────────┐
│  Rust Core            │  #[uniffi::export] pub async fn core_load() -> Result<T, E>
└───────────────────────┘
```

**How it works:**  
UniFFI directly exports Rust `async fn`. The generated Swift binding is a native `async func`. UniFFI internally manages a `RustFuture` handle, polling it on completion, then freeing it. Tokio remains the executor; UniFFI drives the poll/complete/free lifecycle.

**Mechanism:** `#[uniffi::export]` on `async fn`, available since UniFFI 0.28.

---

### Strategy C — Callback-Only Public API (no Swift async)

```
┌───────────────────────┐
│  Swift Shell          │  coreLoad { result in ... }  ← plain closure
└──────────┬────────────┘
           │  UniFFI callback_interface
┌──────────▼────────────┐
│  Rust Core            │  pub fn core_load(cb: Arc<dyn LoadCallback>)
└───────────────────────┘
```

**Mechanism:** `uniffi::export(callback_interface)` — same as Strategy A but without the generated Swift async wrapper.

---

### Strategy D — Job Handle / Polling Model

```
  let handle = coreStartLoad()       // returns JobHandle (UniFFI Object)
  while !handle.isReady() { ... }    // poll from Swift side
  let result = try handle.result()
```

**Mechanism:** `uniffi::Object` with polling methods. No UniFFI async machinery needed.

---

### Strategy E — Blocking / Synchronous Façade

```
  let result = try coreLoadSync()    // blocks the calling thread
```

**Mechanism:** `block_on` wrapping inside the exported Rust function.

## 3. Evaluation Criteria

| Criterion | Weight | Rationale |
|-----------|--------|----------|
| **Idiomatic output** | High | The generated code must feel native to both Rust and Swift developers — not like it was written by a machine. |
| **Generator complexity** | High | Sarah must own the seam. Lower complexity = fewer edge cases, better diagnostics, cleaner output. |
| **Sendable / concurrency safety** | High | Swift 6 strict concurrency makes any `Sendable` gap a hard compile error. |
| **Cancellation fidelity** | Medium | `Task.cancel()` on the Swift side should propagate to Rust where possible. |
| **UniFFI stability** | Medium | The mechanism must be production-ready in UniFFI 0.28+, not experimental. |
| **Tokio compatibility** | Medium | Core owns the executor; the strategy must not leak executor concerns upward. |
| **Debuggability** | Medium | When things go wrong, the generated code must be easy to trace. |
| **Future-proofness** | Low | Should not close off native async migration as UniFFI matures. |

## 4. Ranked Evaluation

### Rank 1 — Strategy A: Three-Zone Hybrid ✅ RECOMMENDED

| Criterion | Score | Notes |
|-----------|-------|-------|
| Idiomatic output | ★★★★★ | Rust side: pure Tokio async. Swift side: pure `async func`. Both are idiomatic. |
| Generator complexity | ★★★★☆ | Sarah generates one callback trait + one spawn + one `withCheckedThrowingContinuation` wrapper. Mechanical and testable. |
| Sendable safety | ★★★★★ | `callback_interface` requires `Send + Sync` on the trait, which UniFFI enforces. No `Sendable` gap. |
| Cancellation fidelity | ★★★☆☆ | `Task.cancel()` can be wired via a `cancel()` method on the callback trait; not automatic but straightforward to generate. |
| UniFFI stability | ★★★★★ | `callback_interface` is the oldest, most battle-tested UniFFI mechanism. |
| Tokio compatibility | ★★★★★ | Tokio executor entirely in Core; nothing leaks upward. |
| Debuggability | ★★★★☆ | The callback boundary is explicit and traceable. |
| Future-proofness | ★★★★☆ | The generator can offer `--async-mode native` to switch to Strategy B once it stabilises. |

**Downsides:**

1. **Generator owns the continuation wrapper.** Sarah must generate, test, and maintain the `withCheckedThrowingContinuation` scaffolding for every async function. This is not free — it is relocation of complexity from the raw UniFFI machinery to Sarah's output layer, where you control it.
2. **Contributor cognitive load.** Developers working on Sarah itself must understand Swift structured concurrency *and* Tokio concurrently, because they are reading and testing generated code from both sides simultaneously.
3. **Cancellation is not automatic.** Unlike native UniFFI async where cancellation propagation is part of the protocol, the three-zone model requires Sarah to generate explicit cancellation support (an additional `cancel()` callback method and a `CancellationToken` in the Rust spawn). It is doable but must be implemented deliberately.
4. **Double closure overhead.** Every async call creates one Swift continuation closure and one Rust `Arc<dyn Callback>` allocation. For high-frequency calls (e.g., streaming), this adds up. A future optimisation path is batching via a channel strategy (see §6).

**Verdict:** Best overall. The downsides are implementation costs that live inside Sarah's generator — they do not leak into the output API surface that Swift or Rust developers see.

---

### Rank 2 — Strategy B: Native UniFFI Async ⏳ STRONG FUTURE OPTION

| Criterion | Score | Notes |
|-----------|-------|-------|
| Idiomatic output | ★★★★★ | The cleanest possible output: `pub async fn` on Rust, `async func` on Swift. |
| Generator complexity | ★★★★★ | Sarah generates almost nothing — just `#[uniffi::export]` on `async fn`. |
| Sendable safety | ★★★☆☆ | UniFFI 0.28 Swift async bindings have documented caveats around incomplete `Sendable` conformances for some generated async scenarios. This is the primary reason it does not rank first today. |
| Cancellation fidelity | ★★★★★ | Cancellation is built into UniFFI's `RustFuture` protocol. |
| UniFFI stability | ★★★☆☆ | Available since 0.28 but still evolving. Swift-side async bindings are newer than callback_interface and carry more open issues. |
| Tokio compatibility | ★★★★★ | Tokio drives the Rust `async fn`; UniFFI handles the rest. |
| Debuggability | ★★★☆☆ | The `RustFuture` lifecycle (poll/complete/free) is less transparent than an explicit callback. |
| Future-proofness | ★★★★★ | This is the direction UniFFI is moving. When the Swift-side caveats resolve, Sarah should default to this mode. |

**Downsides:**

1. **Swift `Sendable` conformance gaps.** UniFFI's own documentation explicitly calls out incomplete `Sendable` support for certain generated async scenarios in Swift 6 strict concurrency mode. This can cause hard build errors in client code.
2. **RustFuture lifecycle complexity.** The poll/complete/free lifecycle exists regardless of whether you see it; it just lives inside UniFFI rather than in Sarah's generator. When something goes wrong, the stack trace crosses multiple abstraction layers.
3. **Less control.** Sarah cannot customise the cancellation strategy, error mapping, or retry logic without forking UniFFI or wrapping the generated binding.

**Verdict:** Will be the default once the Swift concurrency rough edges settle. Sarah should implement it as `--async-mode native` today so migration is a flag flip, not a rewrite.

---

### Rank 3 — Strategy C: Callback-Only Public API ⚠️ MIDDLING

**Downsides:**

1. Exposes raw callback style to SwiftUI — a `ViewModel` cannot call `try await fetchFeed()` naturally.
2. Requires manual callback management in the Shell (capture, retain, release), which is exactly the work Sarah is designed to eliminate.
3. Structured concurrency benefits (cancellation, timeouts, `async let`) are lost.

**When useful:** As a fallback for functions that return multiple values over time (streaming / push patterns). Strategy A can degrade to this shape deliberately.

---

### Rank 4 — Strategy D: Job Handle / Polling ❌ WEAK

**Downsides:**

1. Polling maps poorly to SwiftUI's event-driven model.
2. Requires the Swift Shell to implement a polling loop or a DispatchQueue timer — far from idiomatic.
3. Introduces intermediate `JobHandle` objects that must be memory-managed across the FFI boundary.
4. Sarah would need to generate and maintain the polling interface as well as the actual work function.

**When useful:** Only for work that is genuinely backgrounded with independent observable progress (file download with pause/resume). Not a general async strategy.

---

### Rank 5 — Strategy E: Blocking / Synchronous Façade ❌ AVOID

**Downsides:**

1. `block_on` inside a UniFFI-exported function blocks the thread UniFFI called it on. If that thread belongs to the Swift concurrency cooperative pool, it starves other tasks — a known Swift concurrency hazard.
2. UniFFI's async model exists specifically to avoid this; blocking semantics fight the design.
3. Cannot support cancellation.
4. Any function that takes > ~16 ms will visibly hitch on the main thread.

**When useful:** Only for trivially fast functions that are logically synchronous but implemented with async internally. Use `futures::executor::block_on` sparingly and never on the main thread.

## 6. Recommendation

### Default: Strategy A (Three-Zone Hybrid)

Sarah **must** generate the three-zone scaffold for all `async func` declarations classified as Tier A1 (async-safe) or Tier A2 (async-class):

```rust
// Generated by sarah-cli for: async func fetchFeed()

#[uniffi::export(callback_interface)]
pub trait FetchFeedCallback: Send + Sync {
    fn on_result(&self, result: FeedData);
    fn on_error (&self, error:  String);
    fn on_cancel(&self);                  // cancellation hook
}

#[uniffi::export]
pub fn fetch_feed(callback: Arc<dyn FetchFeedCallback>) {
    let token = CancellationToken::new();
    tokio::spawn(async move {
        tokio::select! {
            _ = token.cancelled() => callback.on_cancel(),
            result = internal_fetch_feed() => match result {
                Ok(data) => callback.on_result(data),
                Err(e)   => callback.on_error(e.to_string()),
            }
        }
    });
}
```

```swift
// Generated by sarah-cli for: async func fetchFeed()

func fetchFeed() async throws -> FeedData {
    try await withCheckedThrowingContinuation { continuation in
        let cb = FetchFeedCallbackImpl { result in
            continuation.resume(returning: result)
        } onError: { err in
            continuation.resume(throwing: SarahError(err))
        } onCancel: {
            continuation.resume(throwing: CancellationError())
        }
        CoreFFI.fetchFeed(callback: cb)
    }
}
```

### Opt-in: Strategy B (Native UniFFI Async)

Sarah **must** implement `--async-mode native` as an explicit opt-in today, so teams tracking UniFFI HEAD can test it. When `--async-mode native` is set, the generated output is:

```rust
#[uniffi::export]
pub async fn fetch_feed() -> Result<FeedData, SarahError> {
    internal_fetch_feed().await.map_err(SarahError::from)
}
```

No Swift wrapper is generated; UniFFI generates the `async func` binding directly.

### Migration Path

When UniFFI resolves the Swift `Sendable` conformance gaps (tracked at [mozilla/uniffi-rs #2097](https://github.com/mozilla/uniffi-rs/issues/2097)), Sarah should:

1. Flip the default from `--async-mode bridge` to `--async-mode native`.
2. Emit a deprecation notice for callback-bridge output.
3. Provide a one-pass migration tool (`sarah migrate-async <dir>`) that rewrites callback-bridge scaffolding to native async in all generated files.

## 7. Diagnostics

The classifier must emit the following diagnostics for async declarations:

| Code | Tier | Severity | Trigger |
|------|------|----------|---------|
| `A1-ASYNC-FUNC` | A1 | Info | `async func` with only Tier 1 param/return types → three-zone scaffold generated |
| `A2-ASYNC-CLASS` | A2 | Warning | `async func` inside a `class` → three-zone scaffold + Arc<Mutex<T>> |
| `A3-ASYNC-ACTOR` | A3 | Error | `actor` declaration → Shell-only; no Core output generated |
| `A4-MAIN-ACTOR` | A4 | Error | `@MainActor` annotation → Shell-only; must not cross FFI |

## 8. Open Questions

1. **Streaming / AsyncSequence.** Strategy A handles request-response cleanly. Streaming (Swift `AsyncSequence` / Rust `tokio::sync::mpsc` channel) needs a separate SPEC (candidate: SPEC-007).
2. **Typed errors across FFI.** The `on_error(&self, error: String)` above uses string serialisation. A richer typed error strategy should be specified before Phase 2c begins.
3. **`withTaskCancellationHandler` integration.** The Swift Shell should detect `Task` cancellation and call a `cancel()` method on the Rust side. This is sketched in §6 but not yet fully specified.
