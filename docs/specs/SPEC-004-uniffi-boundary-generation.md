## Problem Statement

For Shell files (Tier 3) and for any Core type exposed across the Swift/Rust boundary, the transpiler must generate correct UniFFI annotations (proc-macro style) so that:

1. The Swift Shell can call Rust Core functions without writing any FFI glue manually.
2. All types crossing the boundary are correctly serialized/deserialized by UniFFI's scaffolding.
3. Errors are propagated as typed Swift `Error` values, not opaque integers.

---

## Goals

1. Produce `#[uniffi::export]` and `#[derive(uniffi::...)]` annotations on all Core types and functions that cross the boundary.
2. Map Swift types to UniFFI wire types correctly (see SPEC-002 Type Mapping Table).
3. Generate a `uniffi.toml` configuration file for the crate.
4. Support synchronous functions in v1; async support deferred to SPEC-006.

---

## UniFFI Annotation Rules

### Rule 1 — Crate Setup

Every Core crate must include the scaffolding macro at the crate root:

```rust
uniFFI::setup_scaffolding!();
```

And declare in `Cargo.toml`:

```toml
[lib]
crate-type = ["cdylib", "staticlib"]

[dependencies]
uniFFI = { version = "0.28", features = ["scaffolding"] }

[build-dependencies]
uniFFI = { version = "0.28", features = ["build"] }
```

### Rule 2 — Record Types (Tier 1 structs)

```rust
#[derive(Debug, Clone, uniffi::Record)]
pub struct Point {
    pub x: f64,
    pub y: f64,
}
```

All fields must be `pub`. All field types must be in the UniFFI type mapping table.

### Rule 3 — Enum Types (Tier 1 enums)

```rust
#[derive(Debug, Clone, uniffi::Enum)]
pub enum Direction {
    North,
    South,
    East,
    West,
}
```

Associated-value enums are supported when inner types are also UniFFI-compatible.

### Rule 4 — Object Types (Tier 2 classes)

```rust
#[derive(uniffi::Object)]
pub struct Counter {
    value: std::sync::Mutex<i64>,
}

#[uniffi::export]
impl Counter {
    #[uniffi::constructor]
    pub fn new(initial: i64) -> Arc<Self> {
        Arc::new(Self { value: std::sync::Mutex::new(initial) })
    }

    pub fn value(&self) -> i64 {
        *self.value.lock().unwrap()
    }
}
```

`uniffi::Object` types are always passed as `Arc<T>` across the boundary.

### Rule 5 — Error Types

Errors must implement `std::error::Error` (via `thiserror` or manually) and derive `uniffi::Error`:

```rust
#[derive(Debug, thiserror::Error, uniffi::Error)]
pub enum CounterError {
    #[error("Counter overflow")]
    Overflow,
    #[error("Counter underflow")]
    Underflow,
}
```

Throwing functions use `Result<T, ErrorType>`:

```rust
#[uniffi::export]
impl Counter {
    pub fn increment(&self) -> Result<(), CounterError> { ... }
}
```

## uniffi.toml

Each crate that exposes Swift bindings must include a `uniffi.toml` at the crate root:

```toml
[bindings.swift]
# Namespace prefix applied to all generated Swift types
ffi_module_name = "CoreFFI"

[bindings.kotlin]
package_name = "com.example.core"
```

## Generated Swift Binding (Illustrative)

UniFFI auto-generates the following Swift interface from the Rust annotations above. This is illustrative — the actual file is generated by `cargo build`:

```swift
// AUTO-GENERATED — do not edit
public class Counter {
    private let pointer: UnsafeMutableRawPointer

    public init(initial: Int64) { ... }
    public func value() -> Int64 { ... }
    public func increment() throws { ... }
    public func decrement() throws { ... }
    public func reset() { ... }
}
```

## Acceptance Criteria

| # | Scenario | Expected |
|---|----------|----------|
| 1 | Tier 1 `struct` | `#[derive(uniffi::Record)]` emitted |
| 2 | Tier 1 `enum` | `#[derive(uniffi::Enum)]` emitted |
| 3 | Tier 2 `class` | `#[derive(uniffi::Object)]` + `#[uniffi::export]` impl emitted |
| 4 | Throwing function | `Result<T, E>` with `#[derive(uniffi::Error)]` on error type |
| 5 | Crate root | `uniffi::setup_scaffolding!()` present |
| 6 | `Cargo.toml` | `cdylib` + `staticlib` crate types; `uniffi` dependency with `scaffolding` feature |
