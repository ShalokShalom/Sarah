# ADR-006 — Three-Zone Async Boundary Model

**Status:** Accepted  
**Date:** 2026-04-02  
**Deciders:** Sarah Project  
**Related specs:** SPEC-006, SPEC-004, ADR-005

---

## Context

Sarah's architecture has a hard seam between the Rust Core (compiled to a shared library) and the Swift Shell (a SwiftUI application). This seam is mediated by UniFFI. When both sides are synchronous, the bridging is straightforward. When Swift `async func` is introduced, three strategies are available — see SPEC-006 §1 for the full enumeration.

The decision is consequential because it affects:

- Whether `Sendable` conformance warnings appear in the Swift Shell under strict concurrency checking.
- Whether Swift structured concurrency (`async let`, `TaskGroup`) composes with the generated API.
- Whether Rust Core code can remain in Tier 1 (golden-core safe Rust) or must be promoted to async.
- The migration cost when UniFFI resolves its upstream Sendable gap.

---

## Decision

**Adopt the Three-Zone Model as described in SPEC-006.**

The three zones are:

1. **Rust Core** — owns the Tokio executor entirely. No async leaks upward through the UniFFI boundary. Core functions that serve async callers are synchronous at the API level; they spawn Tokio tasks internally and deliver results via callback.

2. **UniFFI Boundary** — uses the stable callback interface pattern (`#[uniffi::export(callback_interface)]`). This is synchronous on the FFI boundary, Sendable-gap-free, and battle-tested in Mozilla production codebases.

3. **Swift Shell** — presents a pure native `async func` surface to SwiftUI. All `withCheckedThrowingContinuation` wrappers are generated by Sarah's transpiler. Shell developers and SwiftUI developers never write or read FFI bridge code.

The three zones use **different async strategies without contradiction** because each zone's strategy is the correct choice for its environment:

- Tokio is the right executor for Rust. Exposing it to Swift would require sharing executor state across the language boundary, which is unsafe and unsupported.
- Callback interfaces are the right UniFFI pattern today because they are stable, `Send`-constrained only on the result type, and produce no `Sendable` warnings.
- `withCheckedThrowingContinuation` is the correct Swift bridge from callback to native async. It is generated, not written by hand, so its verbosity is irrelevant.

---

## Consequences

### Positive

- **No Sendable gap.** The FFI boundary is synchronous; UniFFI's open Sendable issue never surfaces in generated code.
- **Full structured concurrency.** SwiftUI callers use `async let`, `TaskGroup`, and `withThrowingTaskGroup` freely because the Shell surface is native `async func`.
- **Tier 1 Core stays golden.** Using `spawn_blocking` for sync-body functions means the vast majority of transpiled Rust code has no `async` keyword at all.
- **Single-layer migration.** Moving to native UniFFI async (Stage 2) requires only regenerating the Shell bridge. Rust Core and SwiftUI are untouched.
- **Cancellation is automatable.** The `@cancellable` annotation in SPEC-006 §6 gives the transpiler enough information to generate `withTaskCancellationHandler` + a Rust `CancellationToken` pair.

### Negative / Trade-offs

- **`withCheckedThrowingContinuation` in generated code** is verbose. It is never read by humans in normal flow, but it will appear in diffs and generated file previews. Comment headers in generated files will explain this is intentional bridge code, not hand-written.
- **Two async mental models to understand** for contributors working on the transpiler itself. SPEC-006 and this ADR are the canonical references; the onboarding doc (`00_ONBOARDING_THE_CAUSE.md`) links to both.
- **`ASYNC-LOCK-RISK` (Tier A2) requires developer attention.** The codegen can mitigate the deadlock risk (lock-before-await pattern) but cannot eliminate the underlying design tension of async + shared mutable state. Developers must review A2 diagnostics.

---

## Alternatives Considered

### Native Async Only (Strategy 1)

Rejected. UniFFI 0.28's `Sendable` conformance gap produces warnings under `-strict-concurrency=complete`. Apple's direction is toward making strict concurrency the default. Shipping generated code with known concurrency warnings is not acceptable for a project whose primary value proposition is correctness.

### Callback Bridge Only (Strategy 2)

Rejected. Breaks Swift structured concurrency. Produces a Swift API surface that requires manual `withCheckedContinuation` wrapping by the Shell developer, which is exactly the kind of glue Sarah is designed to eliminate.

### Do Nothing / Defer

Rejected. Async is load-bearing for any real-world iOS/macOS application. Deferring it to post-1.0 would make Sarah unusable for production apps.

---

## Review Trigger

This decision should be revisited when the UniFFI upstream Sendable conformance issue closes. At that point, Stage 2 migration (native async, `--async-mode=native` flag) should be evaluated. The three-zone model does not prevent this migration; it explicitly prepares for it.
